# P0 端到端联调与真机平板测试手把手指南

这份文档的目标不是“列出一个抽象 checklist”，而是带你**按顺序一步一步完成一次完整的 P0 联调**。

你只需要照着做，就能验证这套 StudyAgent 在当前仓库里的完整链路是否可用：

- 服务是否能启动
- 管理员是否能通过 `8080` 前端入口正常登录
- 学生账号是否能导入
- 知识库是否能导入，并且能看清系统实际切分了什么
- 学生是否能进行真实 RAG 流式问答，并拿到推荐题卡片
- 统计 / 导出 / 审计 / 监控是否工作
- 平板在真实校园局域网下是否能正常使用

---

## 1. 这次测试要验证什么

这次 P0 联调的核心目标是：

1. **功能完整性**：从“启动服务”到“学生平板上能真正问答”整条链路能跑通。
2. **真实业务闭环**：不是只看接口通不通，而是验证“管理端导入资料 → 学生端提问 → 管理端看到统计”的闭环。
3. **真机可用性**：平板访问、输入、流式输出、弱网场景要能正常工作。
4. **试运行 readiness**：确认这套系统已经具备进入小范围试运行的基本条件。

---

## 2. 测试前你需要准备什么

### 2.1 你需要准备的文件

请提前准备下面 5 份测试资料：

1. `1` 份 `txt` 文档  
   例：一小段物理或数学知识点说明

2. `1` 份 `docx` 文档  
   例：教师整理的讲义

3. `1` 份**可选中文字**的 `pdf`  
   例：教材或试卷 PDF，但必须能在 PDF 阅读器里直接复制文字

4. `1` 份**扫描版 PDF**  
   例：纯图片扫描件，用来验证“扫描版 PDF 检测”

5. `1` 份学生名单 `xlsx`  
   建议至少 3 个学生，表头如下：

| student_no | full_name | grade | class_name | password |
|---|---|---|---|---|
| 20260001 | 张三 | 1 | 1班 | 123456 |
| 20260002 | 李四 | 1 | 1班 | 123456 |
| 20260003 | 王五 | 1 | 1班 | 123456 |

说明：

- `password` 列可以留空；留空时系统会默认使用学号后 6 位
- 表头也可以使用中文：`学号 / 姓名 / 年级 / 班级`

推荐额外再准备 2 份资料，用来覆盖这轮新增验收点：

6. `1` 份题库或习题 `docx`
   例：至少 3 道题，最好带明确题号，且题目后面紧跟答案 / 解析，便于验证按题切分与配对

7. `1` 份带图片的题目 `docx`
   例：几何体、受力图、电路图、实验装置图，用来验证图片抽取和预览是否正确

---

### 2.2 你需要准备的设备

至少准备：

- `1` 台服务器或你的开发机
- `1` 台真机平板
- 平板与服务器处于**同一局域网**

推荐额外准备：

- `1` 部手机用于热点/弱网测试
- `1` 个第二浏览器或无痕窗口，用来同时保持管理员端和学生端登录

---

### 2.3 你需要准备的信息

请提前确认：

- 你有可用的 LLM API Key
- 你知道服务器当前局域网 IP
- 你知道管理员账号和密码

如果还不知道服务器 IP，可以在终端里运行：

```bash
hostname -I
```

通常取输出中的第一个内网地址，例如：

```text
192.168.1.23
```

后面平板访问时会用到它。

## 4. 第二步：准备 `.env` 配置

### 这一步的意义

`.env` 决定系统能不能启动、管理员能不能登录、外部 LLM 能不能调用。  
如果这一步没配好，后面所有测试都会失败。

### 你要做什么

先复制模板：

```bash
cp .env.example .env
```

然后编辑 `.env`。你可以用任意你熟悉的编辑器，例如：

```bash
nano .env
```

或者直接用图形化编辑器打开。

---

### 你至少要修改这些项目

#### 4.1 JWT 密钥

找到：

```env
JWT_SECRET_KEY=change-me
```

改成一个你自己的随机字符串，例如：

```env
JWT_SECRET_KEY=studyagent-p0-test-2026-very-secret
```

**意义**：这是登录签名密钥，不改的话测试环境太弱。

---

#### 4.2 管理员账号

找到：

```env
BOOTSTRAP_ADMIN_USERNAME=admin
BOOTSTRAP_ADMIN_PASSWORD=StudyAgent123
```

你可以保持默认，也可以改成你想要的，例如：

```env
BOOTSTRAP_ADMIN_USERNAME=admin
BOOTSTRAP_ADMIN_PASSWORD=AdminP0Test123
```

**意义**：第一次启动时会自动种一个管理员账号。

---

#### 4.3 主 LLM 配置

找到：

```env
LLM_PRIMARY_NAME=minimax
LLM_PRIMARY_BASE_URL=https://api.minimaxi.com/v1
LLM_PRIMARY_API_KEY=
LLM_PRIMARY_MODEL=MiniMax-M2.7
```

把 `LLM_PRIMARY_API_KEY` 填成你的真实 key，例如：

```env
LLM_PRIMARY_API_KEY=你的真实key
```

**意义**：这是学生真实问答时最关键的配置。没有它，聊天大概率只能走兜底。

---

#### 4.4 备选 LLM 配置

找到：

```env
LLM_FALLBACK_NAME=qwen
LLM_FALLBACK_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_FALLBACK_API_KEY=
LLM_FALLBACK_MODEL=qwen-plus
```

把 `LLM_FALLBACK_API_KEY` 填上，如果你有的话：

```env
LLM_FALLBACK_API_KEY=你的真实key
```

**意义**：主 LLM 异常时，系统可以切到备选，避免联调时因为单一厂商问题卡住。

---

### 这一步完成后你做什么

保存 `.env`，退出编辑器。

你可以快速检查关键项是否已经有值：

```bash
rg -n "JWT_SECRET_KEY|BOOTSTRAP_ADMIN_USERNAME|BOOTSTRAP_ADMIN_PASSWORD|LLM_PRIMARY_API_KEY|LLM_FALLBACK_API_KEY" .env
```

### 你应该看到什么

- 所有关键项都能被搜到
- `LLM_PRIMARY_API_KEY` 不应为空
- 管理员用户名和密码是你记得住的值

## 5. 第三步：启动整套服务

### 这一步的意义

P0 联调要验证的是整条链路，所以推荐一次性把 Compose 环境拉起来，而不是只跑单个服务。

### 你要做什么

在项目根目录执行：

```bash
docker compose up -d --build
```

首次启动可能需要几分钟。

---

### 启动后立刻检查服务状态

输入：

```bash
docker compose ps
```

### 你应该重点看这些服务

- `backend`
- `worker`
- `nginx`
- `redis`
- `postgres`
- `chromadb`
- `prometheus`
- `grafana`

### 你应该看到什么

- 每个服务应该是 `Up` 或类似“运行中”状态
- 如果有服务 `Exited` 或一直 `Restarting`，先不要继续下一步

---

### 如果有服务异常，立刻看日志

你可以分别看：

```bash
docker compose logs --tail=200 backend
docker compose logs --tail=200 worker
docker compose logs --tail=200 nginx
```

如果你想持续看：

```bash
docker compose logs -f backend
```

## 6. 第四步：先跑自动 smoke 检查

### 这一步的意义

这一步是在你点页面之前，先自动确认：

- API 活着
- 前端入口活着
- 管理员登录接口能通
- 监控页面能打开

也就是说，先把最低级的问题排掉。

### 你要做什么

在项目根目录执行下面这条命令：

```bash
API_BASE_URL=http://127.0.0.1:8002 \
WEB_BASE_URL=http://127.0.0.1:8080 \
PROMETHEUS_URL=http://127.0.0.1:9090 \
GRAFANA_URL=http://127.0.0.1:3000 \
ADMIN_USERNAME=你的管理员用户名 \
ADMIN_PASSWORD=你的管理员密码 \
bash scripts/post_deploy_check.sh
```

例如，如果你用的是：

- 用户名：`admin`
- 密码：`AdminP0Test123`

那么命令是：

```bash
API_BASE_URL=http://127.0.0.1:8002 \
WEB_BASE_URL=http://127.0.0.1:8080 \
PROMETHEUS_URL=http://127.0.0.1:9090 \
GRAFANA_URL=http://127.0.0.1:3000 \
ADMIN_USERNAME=admin \
ADMIN_PASSWORD=AdminP0Test123 \
bash scripts/post_deploy_check.sh
```

### 你应该看到什么

终端里出现类似：

```text
[PASS] API /health
[PASS] API /metrics
[PASS] 管理员登录
[PASS] API /api/auth/me
[PASS] 前端 /login
[PASS] Prometheus ready
[PASS] Grafana /login
All checks passed.
```

### 新增一条必须人工补测的代理链路

`post_deploy_check.sh` 会验证 API 与登录接口是否可用，但浏览器实际访问的是：

```text
http://127.0.0.1:8080/login
```

因此你还要额外确认一次 `8080 -> nginx -> backend` 这条链路没有断。  
在终端执行：

```bash
curl -i -s http://127.0.0.1:8080/login | head

curl -i -s -X POST http://127.0.0.1:8080/api/auth/staff/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"你的管理员用户名","password":"你的管理员密码"}'
```

期望：

- 第一个命令返回 `200 OK`
- 第二个命令返回 `200 OK`

如果登录页能打开，但第二个命令返回 `502 Bad Gateway`，说明前端代理层有问题，不要继续后面的浏览器验收。

### 如果失败了怎么办

- `/health` 失败：先看 `backend` 日志
- 管理员登录失败：检查 `.env` 的管理员用户名密码
- `/login` 失败：看 `nginx` 和 `backend` 状态
- `/login` 能打开，但登录提交失败：优先怀疑 `nginx -> backend` 代理失效，先看 `docker compose logs --tail=200 nginx`
- Grafana / Prometheus 失败：可以先继续主功能联调，但要记下来

## 9. 第七步：上传知识库资料

### 这一步的意义

这一部分验证：

- 上传权限是否正常
- Celery / Redis / ChromaDB / 文本提取链路是否正常
- 扫描版 PDF 是否会被正确拦截

### 你要做什么

#### 9.1 打开“知识库”

点击左侧菜单：

```text
知识库
```

#### 9.1.1 先确认这次上传的资料类型选对了

现在知识库上传不只是选学科，还要选资料类型。  
你要按资料真实用途选择：

- 讲义 / 教材类：`知识讲义` 或 `教材`
- 习题 / 例题：`习题例题`
- 题库 / 试卷：`题库试卷`
- 物理学史 / 生活中的物理 / 阅读拓展：`拓展资料`

意义：

- 讲义 / 教材会优先按段落与章节切分
- 习题 / 题库会优先尝试按题号切分，并抽取题干 / 答案 / 解析 / 图片
- 拓展资料通常不强调章节和难度

注意：如果你上传时选错了资料类型，后面再编辑 metadata 虽然能改显示信息，但**不会自动重新切分**；要按新类型重新处理，必须删除后重新上传。

#### 9.2 先上传 `txt`

1. 学科选择一个与你资料对应的学科，例如：

```text
物理
```

2. 点击：

```text
选择资料并上传
```

3. 选择你的 `txt` 文件

#### 9.3 再分别上传

- `docx`
- 可选中文字 `pdf`
- 扫描版 `pdf`

建议顺序：

1. `txt`
2. `docx`
3. 可选中文字 `pdf`
4. 扫描版 `pdf`

如果你准备了题库 / 习题 `docx`，务必再补测 1 次：

5. 题库或习题 `docx`

上传这份资料时请把资料类型切成：

- `习题例题` 或
- `题库试卷`

### 每次上传后你要看什么

上传后页面里会出现：

- 最新任务
- 导入任务列表
- 已上传资料列表
- 查看切分按钮
- 切分结果预览弹窗

你要重点看：

- 状态是否从 `pending/processing` 走到 `completed`
- 是否显示进度条
- 是否有明确的错误信息
- 如果失败，是否先显示教师可理解的中文提示，而不是直接暴露技术异常
- 最新任务摘要里是否出现：
  - 切分方式
  - 片段数
  - 题目数
  - 答案数
  - 解析数
  - 章节数
  - 图片数

### 现在新增一项必须做的检查：点开“查看切分”

对每一份成功导入的资料，至少抽查 1 次：

1. 在“最新任务”“任务列表”或“已上传资料”里点击：

```text
查看切分
```

2. 在弹窗里检查：

- 每个片段的实际内容是什么
- 如果是题库资料，题干 / 答案 / 解析有没有对上
- 图片有没有保留
- 自动识别出的章节 / 小节是否合理
- 如果是题库资料但没有按题拆分，弹窗里是否明确提示“当前按段落切分”

### 对题库 / 习题资料你要重点核对什么

如果你上传的是 `习题例题` 或 `题库试卷`，至少检查下面几项：

1. 是否按题号拆成多条片段，而不是整篇糊成一个大段落
2. `第 X 题` 是否和题干正文对得上
3. 答案是不是跟对了题号
4. 解析是不是跟对了题号
5. 有图的题目是否能在弹窗里直接看到图片
6. 如果题库最后统一放答案，系统是否至少识别出一部分正确配对

### 对讲义 / 教材资料你要重点核对什么

如果你上传的是 `知识讲义` 或 `教材`，重点看：

1. 是否按段落合理切开
2. 章节 / 小节自动识别是不是接近文档原结构
3. 预览里看到的章节是不是你后续希望 RAG 参考的内容结构

### 预期结果

#### 对于 `txt / docx / 可选中文字 pdf`

应当看到：

- `completed`
- 有导入完成提示
- 有切分摘要
- 文档出现在“已上传资料”里
- 可以点开“查看切分”

#### 对于题库 / 习题 `docx`

理想情况下应当看到：

- `completed`
- 摘要里出现“按题目拆分 X 道题”
- 如果文档里有答案 / 解析，摘要里能看到对应数量
- “查看切分”弹窗里能逐题看到题干 / 答案 / 解析 / 图片

如果没有识别出题号，也应该看到明确提示：

- `未识别到稳定题号，当前按段落切分`

#### 对于扫描版 `pdf`

应当看到：

- `failed`
- 错误信息里包含类似：
  - `无可用文本层`
  - `疑似扫描版`

### 这说明什么

说明：

- 资料上传链路正常
- Worker 正常工作
- 向量库写入正常
- 扫描版 PDF 检测正常
- 教师可以看到“系统实际学到了什么”
- 题库切分质量可以人工验收，而不是黑盒

### 如果卡住怎么办

先在终端执行：

```bash
docker compose logs --tail=200 worker
docker compose logs --tail=200 backend
```

## 10. 第八步：用学生账号登录并做第一轮真实问答

### 这一步的意义

这是整套系统里最关键的步骤。  
你要验证的不是“聊天框能发送”，而是：

- 学生能登录
- 能触发 RAG 检索
- 能触发真实 LLM
- 能流式返回
- 能保存历史

### 你要做什么

## 25. 什么时候可以判定 P0 通过

当下面这些都成立时，可以认为这轮 P0 基本通过：

1. `docker compose up -d --build` 后主要服务都正常
2. `scripts/post_deploy_check.sh` 通过
3. 管理员和学生都能登录
4. `8080` 前端入口的登录代理链路正常，不存在登录页能开但提交返回 `502`
5. `txt / docx / 可选中文字 pdf` 能导入成功
6. 扫描版 `pdf` 能明确失败
7. 导入失败时，错误提示对教师可读，不是直接暴露技术异常
8. 上传完成后能看到切分摘要，并能点开“查看切分”
9. 如果测了题库资料，题干 / 答案 / 解析 / 图片配对基本合理
10. 学生端能完成至少 1 次真实流式问答，并能展示推荐题卡片
11. 统计、导出、审计、监控都可用
12. 平板在真实局域网环境中可以完成完整使用流程
13. 没有阻塞级问题（例如无法登录、无法问答、页面打不开、平板完全不可用）

## 26. 如果联调中途卡住，优先看哪里

### 登录有问题

看：

```bash
docker compose logs --tail=200 nginx
docker compose logs --tail=200 backend
```

重点检查：

- `.env` 管理员账号密码
- 浏览器访问的是不是 `http://127.0.0.1:8080/login`
- 登录请求返回码
- 如果登录页能打开但提交失败，执行：

```bash
curl -i -s -X POST http://127.0.0.1:8080/api/auth/staff/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"你的管理员用户名","password":"你的管理员密码"}'
```

- 如果这里返回 `502 Bad Gateway`，优先怀疑 `nginx` 还指向旧后端地址，可先尝试：

```bash
docker compose up -d --build --no-deps nginx
```

---

### 知识库导入有问题

看：

```bash
docker compose logs --tail=200 worker
docker compose logs --tail=200 backend
```

重点检查：

- Redis
- Worker 是否启动
- 是否扫描版 PDF
- “查看切分”弹窗里系统到底切成了什么
- 是否上传时选错了资料类型
- 如果资料类型后改过，是否需要删除后重新上传来触发重新切分

---

### 聊天有问题

看：

```bash
docker compose logs --tail=200 backend
curl -s http://127.0.0.1:8002/metrics | rg "chat_|llm_queue_depth|cache_hit_rate"
```

重点检查：

- LLM key 是否正确
- 是否命中限流 / 队列
- SSE 是否中断

---

### 平板访问不了

先确认：

1. 平板和服务器是否在同一局域网
2. 平板访问的是不是 `http://服务器IP:8080/login`
3. 服务器防火墙有没有挡 `8080`
4. `nginx` 是否正常运行
