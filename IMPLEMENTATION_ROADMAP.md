# StudyAgent 后续推进路线图

本文档承接 `DEVELOPMENT_PLAN.md`，用于把“现状 → 下一步 → 阶段验收 → 需要用户配合的事项”明确化，并作为后续逐阶段推进的执行依据。

## 当前状态

当前仓库已经具备首轮工程骨架：

- 后端：FastAPI、认证、聊天、知识库、统计、智能体配置、审计、Celery 任务骨架
- 前端：登录页、学生答疑页、管理看板、知识库、智能体配置、用户管理页
- 部署：Docker Compose、Nginx、Prometheus、Locust、Alembic 基础目录
- 测试：`pytest` 已有过滤、苏格拉底引导、RAG 基础测试

但距离 `DEVELOPMENT_PLAN.md` 的“可联调 / 可试运行”状态还有几个关键缺口：

1. Redis 仍主要是内存回退，尚未成为真实共享状态基础设施
2. ChromaDB / Embedding 还未完成真实向量检索链路
3. SSE 仍是安全优先的“先生成后分块推送”，不是真正首 token 流式
4. Celery 导入链路尚未完成完整闭环验收
5. 管理端仍缺少任务进度、审计日志等关键页面
6. 缺少真实压测与 SLO 验收

## 推进原则

- 每次只推进一个清晰阶段，阶段完成后暂停并向你汇报
- 若该阶段需要你提供信息、测试环境或手动操作，不跳过，直接明确列出
- 优先推进会影响后续所有阶段的底层能力
- 每个阶段都要求“代码落地 + 自动化验证 + 用户确认”

## 阶段计划

### 阶段 1：Redis 真实接入

目标：把认证会话、登录失败计数、限流等共享状态从纯内存改为 Redis 优先，保留开发回退。

本阶段输出：

- Redis 优先 / 内存回退的统一存储层
- 登录失败计数迁移到 Redis
- 请求限流迁移到 Redis 优先
- 健康检查暴露当前存储后端信息
- Docker Compose / 环境变量补齐 Redis 使用说明
- 新增对应测试

验收标准：

- 无 Redis 时仍可本地开发，不阻塞启动
- 有 Redis 时自动使用 Redis 后端
- `pytest` 通过
- 如本机可拉起 Redis，则补一次真实连通性验证

需要你配合：

- 若你有正式 Redis 地址，可在本阶段结束后提供，用于替换开发默认值并做联调
- 若你希望我直接按容器方式验证，我会在本阶段结束后告知你结果

### 阶段 2：真实 RAG / ChromaDB 链路

目标：把当前轻量回退检索升级为真实 ChromaDB + Embedding 链路。

本阶段输出：

- 查询时按学科过滤并从向量库检索

验收标准：

### 阶段 3：Celery 导入闭环
