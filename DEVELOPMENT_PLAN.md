# StudyAgent 开发计划

## 项目概述

局域网内高中学科 AI 答疑助手，支持千人并发，平板可访问，基于 RAG 检索 + LLM 问答，拒绝闲聊。采用苏格拉底"助产术"教学法，以引导提问启发学生自主思考，而非直接给出答案。智能体具备成长能力，可通过持续投喂资料和调整指令来演进。

## 关键决策

- **学生端访问**：需登录（学号 + 密码），记录提问历史，后台可查看统计数据
- **管理端访问**：JWT 登录，角色区分（admin / teacher）
- **教学理念**：苏格拉底"助产术"——优先反问引导学生思考，而非直接给答案
- **智能体成长**：支持持续投喂资料扩展知识库、调整教学策略提示词、版本化管理智能体行为配置
- **部署模式**：初期单机 Docker Compose；架构上预留水平扩展能力（无状态 Web + 外置 Redis/ChromaDB）
- **Embedding 硬件**：优先尝试 bge-small-zh-v1.5（CPU 可跑）；有 GPU 则升级 bge-m3；查询时只算单条向量，压力可控，批量导入走异步 worker
- **教学输出约束**：允许给出分步解析和思路引导，但禁止直接给出最终结论或标准答案。兜底阶段同样遵守此规则——可以拆解推导过程，但最后一步由学生自己完成。此规则为硬约束，后校验和测试均以此为口径。

## 整体架构

          ├── RAG 检索 (ChromaDB)
          ├── LLM 代理层（排队 + 熔断 + 降级）

## 技术选型

| 模块 | 选型 | 理由 |
|------|------|------|
| 后端框架 | FastAPI + async | 异步高并发 |
| 关系数据库 | SQLite（开发）/ PostgreSQL（生产） | 用户、会话、统计、审计、智能体配置 |
| 向量数据库 | ChromaDB | 轻量、已有经验 |
| LLM 接入 | httpx AsyncClient | OpenAI 兼容接口，直接调用 |
| 缓存/限流 | Redis | 千人并发必须 |
| 异步任务 | Celery + Redis broker | 知识库导入与在线答疑资源隔离 |
| 前端 | Vue 3 + Vite | 响应式，适配平板 |
| 反向代理 | Nginx | 静态资源 + 连接限制 + 负载均衡 |
| 部署 | Docker Compose | 一键部署 |
| Embedding | bge-small-zh-v1.5 (CPU) / bge-m3 (GPU) | 按硬件条件选择 |
| 监控 | Prometheus + Grafana | 可观测性 |

## 核心设计

### 1. 知识库 (RAG)

```
上传资料(PDF/Word/txt) → [Celery Worker] 文本切分 → Embedding → ChromaDB
     用户提问 → 向量检索 Top-K → 拼接上下文 → 苏格拉底引导引擎 → LLM 生成回答
```

- Embedding 使用本地模型，按硬件选择（见关键决策）
- 分科目建 Collection（语文/数学/物理...），检索时按选科过滤
- 文本切分：按段落 + 重叠窗口，chunk_size=512, overlap=64

### 2. 苏格拉底"助产术"教学引擎

核心理念：不直接给答案，而是通过反问引导学生自己推导出结论。

#### 对话策略（多轮状态机）

```
学生提问
   │
   ▼
[判断问题类型] ──概念类──→ 反问："你觉得 X 和 Y 有什么区别？"
   │                        → 等待学生回答 → 追问或肯定 → 补充知识点
   │
   ├──计算/推导类──→ 拆解："这道题的第一步你会怎么做？"
   │                  → 学生尝试 → 纠正或引导下一步 → 最终总结
   │
   └──综合分析类──→ 引导："我们先从哪个角度分析这个问题？"
                    → 逐步搭建思维框架 → 学生补全 → 总结
```

#### 引导深度控制

- **初始引导**：先反问，不给任何答案线索
- **适度提示**：学生尝试后仍困难 → 给出方向性提示（但不直接揭示答案）
- **兜底机制**：多轮（>=3轮）引导后学生仍无法推进 → 给出分步解析（拆解推导过程），但保留最后一步让学生自己完成，不直接给出最终结论或标准答案
- 每轮对话记录引导阶段，存入会话上下文

#### 系统提示词（核心）

```
你是一位高中学科答疑导师，采用苏格拉底"助产术"教学法。

核心原则：
1. 永远不要直接给出最终结论或标准答案。你的任务是通过提问引导学生自己发现答案。
2. 收到学生问题后，先判断其理解程度，然后用反问引导思考方向。
3. 当学生回答正确时，给予肯定并追问更深层的"为什么"。
4. 当学生回答错误时，不要说"不对"，而是用新的问题让学生发现矛盾。
5. 经过充分引导（至少2-3轮对话）后学生仍无法推进，给出分步解析帮助理解推导过程，但保留最后一步让学生自己得出结论。绝不直接给出最终答案。

示例对话：
学生："为什么铁在潮湿空气中会生锈？"
你："好问题！你觉得'生锈'这个过程中，铁发生了什么变化？是物理变化还是化学变化？"
学生："化学变化。"
你："对！那你知道铁在这个化学变化中和什么物质发生了反应吗？提示：潮湿空气中有哪些成分？"

你只解答高中阶段的语文、数学、英语、物理、化学、生物、政治、历史、地理问题。
非学科问题一律回复："抱歉，我只能解答高中学科相关问题，请重新提问。"
```

### 3. 话题过滤（三层守卫）

**第一层（前置分类器）**：独立轻量分类器，在调 LLM 之前判断是否为学科问题。
- 方案 A：基于 fasttext/小型 BERT 的二分类模型（学科问题 vs 非学科）
- 方案 B：关键词白名单 + 正则规则（快速实现，后续可升级）
- 不通过则直接返回拒绝，不消耗 LLM token

**第二层（系统提示词）**：已融入苏格拉底引导提示词中（见 §2）。

**第三层（回答后校验）**：对 LLM 输出做关键词/规则扫描，拦截漏网的非学科内容；同时校验是否违反"不直接给答案"原则。

**对抗测试**：维护一组对抗样例测试集（prompt 注入、角色扮演诱导、RAG 污染等），CI 中自动回归。

### 4. 容量规划与并发设计

#### 目标拆解

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 同时在线连接数 | 1000+ | SSE 长连接，Nginx 层控制 |
| 同时生成请求数 | 50-100 | 受 LLM API QPS/TPM 限制 |
| 部署模式 | 单机 | 初期；预留多机扩展 |
| LLM QPS 预算 | 环境配置项 | 按账号实测和套餐配置，写入 `LLM_MAX_QPS` 环境变量，排队/背压从此读取 |
| p95 首 token 延迟 | < 3s | 含检索 + LLM 首包 |
| p95 完整回答时延 | < 30s | 流式输出，用户感知更短 |

#### 排队与背压

- 请求入 Redis 有界队列（max_queue_size 可配）
- 队列满时返回 HTTP 503 + 友好提示"当前排队人数较多，请稍后重试"
- 前端显示排队位置

#### 超时与取消

- 用户关闭页面 → 服务端检测 SSE 断连 → 取消进行中的 LLM 请求
- LLM 调用设 30s 超时，超时返回"回答生成超时，请重试"

#### 熔断与降级

- LLM API 连续失败 N 次 → 熔断，自动切换备用厂商（MiniMax → Qwen）
- 所有厂商不可用 → 降级为维护模式，显示"AI 答疑服务暂时不可用，请稍后再试"，不暴露原始 RAG 片段（纯 RAG 展示会绕过教学约束和输出校验）
- 熔断状态定期探测恢复

#### SSE 连接管理

- Nginx `limit_conn` 限制单 IP 并发 SSE 连接数（默认 3）
- 心跳保活（15s interval），客户端自动重连
- 服务端追踪活跃连接数，暴露为 Prometheus 指标

### 5. 认证、鉴权与上传安全

#### 角色与权限

| 角色 | 认证方式 | 权限 |
|------|---------|------|
| 学生 | 学号 + 密码登录 | 答疑问答、查看自己的历史 |
| 教师 | JWT 登录 | 上传资料、查看知识库、查看本班学生统计 |
| 管理员 | JWT 登录 | 全部管理功能、用户管理、全校统计、智能体配置、系统配置 |

#### 学生账号管理

- 支持批量导入（Excel 学号名单）
- 初始密码 = 学号后 6 位，首次登录强制改密
- 账号按班级/年级组织，便于统计聚合
- 无自助注册功能，所有学生账号由管理员/教师批量导入

#### 班级归属与数据隔离

- 学生模型包含 `class_id`（班级）和 `grade`（年级）字段
- 教师绑定所管班级列表（`teacher_classes` 多对多关系）
- 教师统计接口自动按绑定班级过滤，仅返回本班数据
- 管理员无隔离限制，可查看全校数据
- 接口鉴权粒度：中间件校验角色 → 服务层按 `class_id` 过滤查询结果

#### 登录安全策略

| 项目 | 规则 |
|------|------|
| 密码存储 | bcrypt 哈希，cost factor >= 12，明文永不落库/日志 |
| 登录失败限速 | 同一账号连续失败 5 次 → 锁定账号 15 分钟；按 `account_id + client_ip` 维度每分钟最多 10 次登录尝试，不做纯 IP 硬锁定（校内共享出口 IP 易误伤） |
| JWT 有效期 | access_token 30 分钟，refresh_token 7 天 |
| Token 刷新 | refresh_token 携带 `jti` + `family_id`。Redis 两层索引：1) `refresh:{user_id}:{family_id}` → 当前有效 jti；2) `user_families:{user_id}` → 该用户所有活跃 family_id 集合。轮换时原子写入新 jti + 失效旧 jti；若已失效 jti 被重放，撤销整个 token family（疑似泄露） |
| access_token 全量失效 | 不逐一枚举黑名单。用户表维护 `password_changed_at` 时间戳，JWT 校验时比对 `iat < password_changed_at` 则拒绝。登出场景仅需黑名单当前单个 access_token |
| 首次登录 | 强制修改初始密码，未改密前拦截所有非改密接口 |
| 密码强度 | 最少 8 位，需包含字母和数字（校内场景不做过度复杂要求） |
| 密码重置 | 仅管理员/教师可操作，重置后：1) 标记为"需改密"状态 2) 更新 `password_changed_at`（已签发 access_token 自动失效）3) 遍历 `user_families:{user_id}` 撤销所有 refresh token family → 强制踢出全部会话。操作写入审计日志 |
| 登出 | 撤销当前 refresh token family：原子删除 `refresh:{user_id}:{family_id}` 键 + 从 `user_families:{user_id}` 集合移除该 family_id；将当前 access_token 加入 Redis 黑名单（TTL = token 剩余有效期） |

#### 上传安全

- 文件类型白名单：`.pdf`, `.docx`, `.txt`（后续按需加 `.xlsx`, `.pptx`）
- MIME type 校验（不仅看后缀）
- 单文件大小限制：50MB
- 文件解析在独立 Celery worker 中执行，与 Web 进程资源隔离
- 解析超时限制：单文件 5 分钟
- 上传文件存储在隔离目录，Web 进程无直接执行权限

#### 操作审计

- 所有管理操作（上传/删除知识库、用户管理、配置变更）写审计日志
- 审计日志字段：时间、操作者、操作类型、目标对象、结果、IP
- 日志持久化到 SQLite 或文件，可通过管理后台查看

### 6. 异步任务队列（知识库导入）

知识库导入（上传/切分/Embedding/索引重建）全部走 Celery 异步任务，与在线答疑完全隔离。OCR 暂不纳入首期范围；如后续需要支持图片/扫描件，再补 OCR 引擎选型、图片格式白名单和资源预算。

**扫描版 PDF 检测**：学校场景中扫描版（纯图片）PDF 很常见，这类文件能通过上传校验但无法有效提取文本。Celery worker 在文本提取阶段做组合判断：1) 检查 PDF 是否存在文本层（无文本层 → 纯图片型）；2) 有文本层时计算字符密度（提取字符数 / 页数），密度极低（阈值可配，经验初始值 50 字符/页）且页数 > 1 → 疑似扫描版。命中任一条件则标记任务失败（不进入重试链路、不生成空索引），返回明确失败原因。管理端任务列表展示："该 PDF 疑似扫描版，无法有效提取文本。请上传可选中文字的 PDF，或将内容转为 Word/TXT 后重新上传。后续版本将支持 OCR。"单页短文档（如公式表、简答题）因页数 = 1 不会被误判。

```
管理端上传文件 → FastAPI 接收存储 → 创建 Celery 任务 → 返回 task_id
                                                          │
Celery Worker（独立进程/容器）:                              │
  文件解析 → 文本切分 → Embedding → 写入 ChromaDB           │
                                                          │
管理端轮询 task_id → 获取进度（0%~100%）/ 成功 / 失败       ←┘
```

- 任务失败自动重试（最多 3 次，指数退避）
- 支持取消进行中的任务
- 前端显示实时进度条
- 批量导入时限制并发 worker 数，避免打满 CPU

### 7. 学生提问统计与数据分析

#### 数据采集（每次对话自动记录）

| 字段 | 说明 |
|------|------|
| student_id | 学生 ID |
| subject | 学科 |
| question_text | 原始提问 |
| conversation_turns | 对话轮数（反映引导深度） |
| guidance_stage | 最终引导阶段（初始引导/适度提示/兜底解析） |
| resolved | 学生是否标记"已解决" |
| created_at | 提问时间 |
| duration_seconds | 对话持续时长 |

#### 后台统计看板（管理端）

- **全校概览**：日/周/月提问量趋势、学科分布、高峰时段
- **班级对比**：各班提问量、学科偏好、平均对话轮数
- **学生个人画像**：提问历史、薄弱学科、引导有效率（自主解决 vs 兜底解析的比例）
- **热点问题**：高频提问聚类，辅助教师发现教学薄弱点
- **导出**：支持 Excel 导出统计报表

### 8. 智能体成长机制

智能体不是一次性配置，而是可以持续演进的系统。

#### 成长维度

| 维度 | 机制 | 操作方式 |
|------|------|---------|
| **知识广度** | 持续投喂资料，扩展 RAG 知识库 | 管理端上传新文档 → Celery 处理 → ChromaDB 增量更新 |
| **教学策略** | 调整苏格拉底引导提示词和策略参数 | 管理端"智能体配置"页面，修改后即时生效 |
| **学科深度** | 按学科细分 Collection，针对性补充 | 管理端按科目管理知识库 |
| **行为规范** | 更新过滤规则、对抗样例 | 管理端更新规则 + CI 回归测试 |

#### 智能体配置版本化

```
agent_config (SQLite 表)
├── id, version
├── system_prompt          # 核心提示词（苏格拉底策略）
├── guidance_params        # 引导参数（最大引导轮数、兜底阈值等）
├── subject_prompts        # 各学科定制提示词（JSON）
├── filter_rules           # 过滤规则
├── created_by, created_at
└── is_active              # 当前生效版本
```

- 每次修改创建新版本，可随时回滚
- 管理端可对比不同版本的配置差异
- 未来可按 A/B 测试分流不同配置

#### 成长闭环

```
学生使用 → 统计数据发现问题 → 管理员调整策略/补充资料 → 智能体行为改善 → 循环
     ↑                                                              │
     └──────────────────────────────────────────────────────────────┘
```

### 10. 可观测性

#### 监控指标（Prometheus）

| 指标 | 类型 | 说明 |
|------|------|------|
| `chat_request_total` | Counter | 答疑请求总数 |
| `chat_first_token_seconds` | Histogram | 首 token 延迟 |
| `chat_full_response_seconds` | Histogram | 完整回答时延 |
| `sse_active_connections` | Gauge | 活跃 SSE 连接数 |
| `sse_disconnect_total` | Counter | SSE 异常断连数 |
| `rag_retrieval_hit_rate` | Gauge | 检索命中率（Top-K 中有相关结果的比例） |
| `llm_api_error_total` | Counter | 外部 LLM API 错误数（按厂商/错误码） |
| `llm_api_latency_seconds` | Histogram | LLM API 调用延迟 |
| `llm_queue_depth` | Gauge | 排队请求数 |
| `cache_hit_rate` | Gauge | 热点问题缓存命中率 |
| `filter_blocked_total` | Counter | 话题过滤拦截数 |
| `guidance_stage_total` | Counter | 各引导阶段计数（初始/提示/兜底） |
| `student_resolved_rate` | Gauge | 学生自主解决率 |
| `celery_task_duration_seconds` | Histogram | 知识库导入任务耗时 |

#### 验收 SLO

| SLO | 目标 |
|-----|------|
| p95 首 token 延迟 | < 3s |
| p95 完整回答时延 | < 30s |
| SSE 异常断连率 | < 1% |
| 话题过滤准确率 | > 95%（对抗测试集） |
| 可用性（月） | > 99.5% |

#### 工具

- Prometheus：采集指标
- Grafana：仪表盘可视化
- 应用日志：结构化 JSON，含 request_id 串联全链路
